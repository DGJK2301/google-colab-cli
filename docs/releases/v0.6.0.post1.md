# google-colab-cli v0.6.0.post1

Release date: 2026-07-20

Baseline: `514db7e032a3e93bba9586cab8fcf00d37d1dd96`

This is a deliberately narrow reliability release. It does not add resource
monitoring, allocation retry loops, JSON result envelopes, model-specific
training logic, or cross-VM scheduling. It closes failures that can make a
training controller hang, request the wrong accelerator, lose an interactive
stdin reply, or install a different Jupyter transport than the one validated in
the source checkout.

## Fixed contracts

### Published dependency identity

Both `pyproject.toml` and `uv.lock` resolve the Colab-specific Jupyter transport
to the exact commit:

```text
https://github.com/googlecolab/jupyter-kernel-client.git
@f18e982c3265df5e923aa9def101ab3fd737e139
```

The published metadata also enforces `pyzmq>=26.0.0`,
`jsonschema>=4.26.0`, and `rpds-py>=0.25.0` so installation into an existing
Jupyter/ML environment cannot silently retain the stale combinations observed
to fail at CLI import time. The console entry point now imports the full CLI
lazily and emits a compact repair message for recognized dependency/ABI
failures.

### Finite execution timeouts

The pinned transport can repeatedly call `Event.wait(0)` after a finite
execution reaches its deadline. The local compatibility guard permits one
already-queued boundary message, then raises `TimeoutError`; continuous output
cannot keep a zero-timeout loop alive indefinitely. The guard is intentionally
local and does not copy the upstream WebSocket receive loop.

`colab exec` and `colab run` now:

- reject zero, negative, NaN, and infinite timeout values before contacting
  Colab;
- pass the same timeout to the `/content` prelude and user-code execution;
- return exit code `124` when the local wait expires;
- state explicitly whether the existing remote kernel may still be running or
  whether ephemeral-runtime cleanup will be attempted.

A timeout is not evidence of remote cancellation. Long work should use
`submit`, `tail`, and `wait`, whose state is materialized in the VM and can be
reattached from another local CLI process.

### Jupyter stdin replies

When the kernel sends `input_request`, the hook now sends exactly one
`input_reply` through the transport's public `input()` method. Custom hooks
receive the full request object and, following the upstream contract, own the
reply send; their return value is not interpreted as a second reply. Password
prompts use `getpass`, and their values are redacted from history events.

### Fail-closed accelerator selection

GPU and TPU strings are parsed by one shared exact mapping used by both `new`
and `run`. Unknown values and simultaneous `--gpu`/`--tpu` requests fail
locally with exit code `2`; they never fall back to A100 or V6E1 and never call
the assignment API.

### Honest allocation failures

`Client.assign()` preserves the original HTTP status and response body. For
backward compatibility, HTTP 412 still raises the deprecated
`TooManyAssignmentsError`, but that type now subclasses `ColabRequestError` and
carries the original evidence. Because 412 can represent usage limits,
entitlement, capacity, or assignment-count constraints, command output does not
present the historical class name as a confirmed diagnosis. Messages for
400/401/403/412/429/503 are concise and avoid raw tracebacks.

### Authentication identity

The default remains the public OAuth2 flow, matching the actual historical CLI
behavior. `DEFAULT_AUTH_PROVIDER` is shared by the CLI callback, state object,
and credential facade. ADC remains available explicitly with `--auth=adc`.
Detached keep-alive processes now enter through the dependency-diagnostic
entry point.

## Code-review hardening

The reviewed patch also serializes temporary timeout-guard installation across
concurrent callers and retains local session recovery state whenever backend
unassignment cannot be confirmed. Cleanup messages distinguish a confirmed
release from an ambiguous network/backend result. Hatch VCS uses its documented
`fallback-version` option so source builds outside a Git checkout resolve to
`0.6.0.post1`.

The update checker now reads the audited fork's GitHub release identity. Its
manual hints and `colab update --install` use an exact fork tag rather than the
upstream PyPI distribution, preventing an update from silently replacing the
reviewed transport and lifecycle fixes.

## Validation

The release adds deterministic tests for:

- deadline expiration, queued-message and event-boundary races;
- stdin, password redaction, custom hooks, and missing input APIs;
- invalid timeouts before session resolution or VM allocation;
- prelude/body timeout exit code and cleanup behavior;
- strict accelerator parsing and HTTP 412/503 presentation;
- preservation of the legacy HTTP 412 exception contract and original evidence;
- published dependency and lock-file identity;
- the OAuth2 default and detached entry point.

A release candidate is complete only when its cumulative patch, source archive,
test report, and SHA-256 checksums are generated from this baseline and verified
by reapplication to a clean worktree. Live Colab CPU/GPU acceptance remains a
separate release gate because it requires user credentials and consumes an
external runtime.

## Known boundaries

- Detached jobs survive local CLI disconnects, not Colab VM reclamation.
- This release does not expose GPU model, VRAM, compute-unit balance, or machine
  shape as structured data.
- It does not automatically retry scarce accelerator allocation.
- It does not implement a cross-VM queue or scheduler.
- The exact Git transport dependency makes this a tag/source/wheel release; a
  conventional PyPI release should wait for an equivalent published transport
  version without a direct VCS requirement.
- Bulk datasets should remain in Drive/GCS; CLI transfer remains the control
  plane for source bundles, checkpoints, and diagnostics.
